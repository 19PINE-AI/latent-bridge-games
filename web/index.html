Latent Bridge

Fast/slow model coupling for real-time agents, demonstrated on Atari. GitHub · Full results

A frozen real-time multimodal model (MiniCPM-o 4.5, 9 B) handles per-frame reactive control at 15 Hz; a frozen reasoning model (Qwen3-VL-8B-Thinking, 8 B) emits strategic guidance at 1 Hz. The two are coupled via a trainable latent bridge: the slow model produces 8 latent tokens in the fast model's input embedding space, prepended to the per-frame sequence so all 36 LLM layers attend over them. This is the same architectural pattern that LLaVA, BLIP-2, Flamingo, and MiniCPM-o itself use for multimodal coupling.

The latent channel beats a text-only baseline by +54 % on Ms. Pac-Man and +26 % on Seaquest. The bridge fails on Space Invaders, and we trace that failure to a methodology issue (the action head's out-of-distribution brittleness, not the bridge itself).

What the demo shows

The slow model's text emission is overlaid as a side panel in each demo. You can watch the latent agent (L) react to the slow's reasoning in real time. The fast-only agent (F) plays without that input — it sees the same frames but has no strategic context.

On Ms. Pac-Man, watch the agent pick a coherent direction once the slow has emitted; on Seaquest, watch it lock into a deterministic kill pattern.

How it works

The slow model thinks (with chain-of-thought) and emits 8 latent vectors per cycle. A small MLP (~33 M trainable params) projects those into the fast model's input embedding space; they're prepended to the fast model's per-frame prompt. All other parameters of both base models stay frozen.

  • Fast: MiniCPM-o 4.5 (9 B), bf16, ~18 GB VRAM
  • Slow: Qwen3-VL-8B-Thinking, bf16, ~17 GB VRAM
  • Bridge bandwidth: 8 tokens × 4096 dims per emission (the bandwidth sweet spot — both N=4 and N=16 underperform)
  • Training: KL(student ‖ teacher) where the teacher is T (text) and the student is L (latent); ~10 min per game on a single GPU
